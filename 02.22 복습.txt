테스트 코드 작성 시 알아두면 좋을 "Golden Rules"
리팩터링을 마친 설계자님이 앞으로 테스트를 짤 때 무기가 될 5가지 팁

1. AAA 패턴을 지키세요
모든 테스트는 세 단계로 구성될 때 가장 아름답습니다.

Arrange (준비): 객체 생성, Fixture 설정.

Act (실행): 실제 메서드 호출 (예: update, stop).

Assert (검증): 결과 확인.
(팁: 코드 사이에 한 줄 공백을 두어 이 단계를 시각적으로 분리하세요.)

2. 한 테스트에서는 '한 가지'만 검증하세요
하나의 def 안에 assert가 너무 많으면, 어디서 왜 틀렸는지 찾기 힘듭니다. "하나의 행동 = 하나의 테스트 함수" 원칙을 지키면 테스트 명칭도 자연스럽게 명확해집니다.

3. Epsilon(오차 범위)을 잊지 마세요
부동 소수점(float)을 다루는 물리 시뮬레이션에서 == 비교는 위험합니다.
assert t.alt == 10.0 대신 assert abs(t.alt - 10.0) < EPSILON 또는 pytest.approx(10.0)을 사용하세요.

4. Boundary Testing (경계값 테스트)
망원경의 가동 범위가 0~90도라면, 딱 90.0도일 때와 90.1도일 때를 테스트하세요. 대부분의 버그는 이 아슬아슬한 경계선에서 발생합니다.

5. Determinism (결정론적 테스트)
테스트는 언제 실행해도 같은 결과가 나와야 합니다. time.sleep() 대신 우리가 제어할 수 있는 dt를 수동으로 주입하여 시간을 조작하는 지금의 방식이 아주 올바른 방향입니다.

test_telescope.py
1. 테스트 케이스 설계 분석
Test 1 (엣지 케이스 처리): 현재 위치와 목표 위치가 동일할 때 불필요한 연산이나 상태 전환 없이 IDLE을 유지하는지 확인합니다. 
    이는 제어 루프에서 발생할 수 있는 '미세 진동(Jitter)'이나 불필요한 가속을 방지하는지 확인하는 필수적인 테스트입니다.

Test 2 (핵심 기능 및 물리 법칙): 목표 지점에 도달했을 때 정밀도(EPSILON) 이내로 들어오는지, 그리고 가장 중요한 
    **"도달 후 속도가 0인가"**를 확인합니다. 이는 관성이나 오버슈트(Overshoot) 처리가 잘 되어 있는지를 검증합니다.

Test 3 (안전 및 예외 상황): stop() 명령이 다른 물리 업데이트보다 우선순위가 높은지(Preemptive), 
    그리고 한 번 정지된 후에는 명시적인 명령 없이 위치가 변하지 않는지(Freeze) 확인하는 안전 로직 검증입니다.

test_command.py
1. 테스트 설계 및 아키텍처 분석
FakeTelescope의 역할: 실제 망원경 하드웨어나 복잡한 물리 엔진 없이도 MoveCommand의 논리 구조를 검증할 수 있게 해줍니다. 
    특히 _target_reached나 _stopped 같은 플래그를 수동으로 조작하여 결정론적인(Deterministic) 테스트 환경을 만든 점이 좋습니다.
상태 기반 테스트: CMD_SUCCESS와 CMD_ABORTED라는 결과 상태를 검증함으로써, 커맨드 객체가 망원경의 상태를 올바르게 관찰(Observe)하고 있는지 
    확인합니다.

2. 주요 테스트 포인트 분석
✅ Test 1: 위임(Delegation) 검증
cmd.execute(fake)가 호출되었을 때, 커맨드가 직접 일을 하는 것이 아니라 망원경(fake)에게 
    enqueue_move를 시키는지 확인합니다. 이는 객체 지향의 책임 분리가 잘 이루어졌는지 확인하는 핵심 테스트입니다.

✅ Test 2: 조건부 성공 로직
주석에 적어주신 "두 번의 update" 로직이 인상적입니다.
첫 번째 update: 명령이 전달되었음을 인지 (has_moved = True).
두 번째 update: 목표 도달 확인 후 최종 성공 처리.
이러한 단계적 검증은 망원경이 움직이기도 전에 "도달 성공"으로 오판하는 레이스 컨디션(Race Condition)을 방지하는 설계로 보입니다.

✅ Test 3: 중단(Abort) 처리
하드웨어 제어에서 가장 중요한 것은 안전입니다. 망원경이 외부 요인(긴급 정지 등)으로 멈췄을 때, 
    커맨드가 무한 루프에 빠지지 않고 즉시 CMD_ABORTED 상태로 전환되는지를 확인하는 필수적인 안전장치 테스트입니다.

test_command_manager.py
1. 테스트 설계 및 아키텍처 분석
상태 관리자로서의 Manager: Telescope가 근육이고 Command가 동작 지시서라면, CommandManager는 이를 스케줄링하는 지휘자입니다. 
    테스트에서 current와 queue를 분리하여 검증한 점이 매우 적절합니다.
Decoupling (결합도 낮추기): FakeCommand에 priority와 scheduled_at 필드를 미리 추가해두신 점이 인상적입니다. 
    이는 향후 우선순위 큐나 예약 실행 기능을 도입할 때 테스트 코드를 크게 수정하지 않아도 되도록 배려한 설계로 보입니다.

2. 주요 테스트 케이스 분석
✅ Test 1 & 2: 즉시 실행 vs 대기(Pending) 로직
시스템이 한 번에 하나의 작업만 수행하도록 보장(Single Active Command)하는지 확인합니다. 
    하드웨어 제어에서 리소스를 선점당하지 않도록 보호하는 가장 기초적이면서 중요한 로직입니다.

✅ Test 3: 연속 실행 (Chain Execution)
cmd1 종료 후 cmd2가 자동으로 실행되는 과정을 테스트합니다.
주목할 점: 여기서도 update를 두 번 호출하는 구조를 사용하셨는데, 이는 매니저의 메인 루프가 
    **[종료 감지] -> [다음 명령 인출]**의 단계를 거친다는 내부 설계를 반영합니다.
이미지 설명: 아래 다이어그램처럼 매니저는 루프를 돌며 현재 상태를 확인하고 큐를 비웁니다.

✅ Test 4: 전역 중단 (Panic/Stop)
단순히 현재 동작만 멈추는 것이 아니라, **대기 중인 모든 예약 작업을 폐기(Clear Queue)**하는 로직을 검증합니다. 
    이는 하드웨어 안전사고 방지를 위한 필수적인 구현입니다.

test_system_controller.py
1. 테스트 설계 및 아키텍처 분석
중앙 집중식 제어 (Centralized Control): SystemController가 등록된 모든 CommandManager를 일괄 통제하는 
    'Master Switch' 역할을 잘 수행하는지 확인합니다. 이는 분산된 하드웨어를 한 번에 제어해야 하는 복잡한 
    시스템(예: 거대 망원경의 경통, 돔, 포커서 동시 제어)에서 필수적인 구조입니다.
안전 우선 정책 (Safety-First): 특히 시나리오 2는 하이브리드 시스템에서 흔히 발생하는 
    **경합 조건(Race Condition)**을 정책적으로 어떻게 정의했는지 보여줍니다.

2. 주요 시나리오 상세 분석
✅ 시나리오 1: 전파(Propagation) 테스트
이 테스트는 SystemController가 단순한 컨테이너가 아니라, 제어 신호의 라우터임을 증명합니다.
분권화된 실행: 각 매니저(A, B)가 독립적인 하드웨어(telescope)를 제어합니다.
수직적 전파: system.global_stop() 호출 한 번이 Manager를 거쳐 최하단 Telescope의 물리 상태까지 관통(Flow-down)하는지를 검증합니다.
의의: 이 테스트가 통과된다면, 하드웨어가 10개로 늘어나도 단 하나의 명령으로 전체 시스템을 
    안전하게 셧다운할 수 있다는 신뢰성을 확보하게 됩니다.

✅ 시나리오 2: 원자성 및 정책 우선순위 (Stop Priority)
이 부분은 설계자의 운영 철학이 반영된 아주 뛰어난 테스트 케이스입니다.
상충 상황: 명령이 성공(SUCCESS)하려는 찰나에 정지(STOP)가 들어온 상황입니다.
해결책: "성공으로 처리해도 무방할 만큼 가깝지만, 정지 명령이 내려졌다면 결과는 무조건 ABORTED여야 한다"는 보수적 안전 원칙을 선택하셨습니다.
의의: 하드웨어 제어에서 '거의 다 왔으니 성공'이라는 타협은 위험할 수 있습니다. ABORTED로 처리함으로써 
    운영자가 "이 명령은 끝까지 완수되지 못했다"는 것을 명확히 인지하게 만듭니다.
